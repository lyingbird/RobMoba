## 第一章 功能概述

### 1.1 需求背景

当前项目已实现基础的1v1 PvP对战（MatchSystem），但存在以下问题：
- **无等待机制**: 玩家进入服务器后各自选英雄，没有"等双方都准备好再开始"的流程
- **英雄选择不同步**: 每个客户端独立执行 HeroSelectUI，对方不知道你选了什么
- **无匹配状态管理**: 没有"等待中/选人中/对战中/结算中"的完整游戏状态机
- **结算后无法重开**: 一局结束后没有"再来一局"或"返回大厅"的机制

这些问题在**两台不同电脑**联网对战时会导致体验割裂。

### 1.2 目标

将项目从"同服本地PvP Demo"升级为**两台电脑真正联网对战**的完整体验：
1. 两个玩家进入同一 Roblox 服务器，能看到彼此
2. 有明确的"等待对手→选英雄→对战→结算"完整流程
3. 所有操作（移动、技能、普攻）在两台电脑上实时同步
4. 代码架构支持后续扩展到 3v3（6人）

### 1.3 范围边界

**本次做**:
- ✅ 服务端游戏状态机（Waiting → HeroSelect → Battle → Result）
- ✅ 等待对手加入机制
- ✅ 同步英雄选择流程（双方都选好再进入对战）
- ✅ 对战倒计时开始
- ✅ 结算界面（胜负 + 数据统计）
- ✅ 重新匹配 / 离开机制
- ✅ 3v3 架构预留（所有接口支持 N 人）

**本次不做**:
- ❌ 跨服务器匹配（TeleportService）— 当前阶段 2 人直接进同服
- ❌ 大厅系统 / 房间列表 — 留给 3v3 阶段
- ❌ Ban/Pick 选人机制 — 留给英雄池足够大时
- ❌ 好友邀请系统
- ❌ 排位 / 段位系统
- ❌ 观战系统
- ❌ 地图改动（使用现有地图）

> 💡[假设] 匹配方式：当前阶段使用"同服务器自动匹配"，2人进入同一 Roblox Place 即开始流程，不引入大厅/房间系统
> 💡[假设] 英雄选择：允许双方选同一英雄（Ban/Pick 机制留给3v3阶段）
> 💡[假设] 地图：使用现有地图，不改动
> 💡[假设] 等待超时：等待第2人加入超时60秒，超时后提示"等待对手中..."持续等待

### 1.4 设计原则

| 原则 | 说明 | 参考 |
|------|------|------|
| **服务端权威** | 所有游戏逻辑（匹配状态、英雄锁定、胜负判定）由服务端控制，客户端只展示 | Roblox 最佳实践 |
| **状态机驱动** | 用明确的状态机管理游戏流程，禁止用散落的 flag 控制流程 | 王者荣耀：大厅→BP→加载→对战→结算 |
| **N人架构** | 所有数据结构使用数组/字典而非硬编码 2 人，支持后续扩展 | 3v3 前置需求 |
| **最小改动** | 尽量复用现有系统（CombatUtils、技能系统、输入系统），只改造匹配和流程 | 降低风险 |

## 第二章 功能详细设计

### 2.1 游戏流程总览

```
玩家加入服务器
    ↓
┌──────────────────────────────────────────────────────────┐
│  阶段1: WAITING（等待玩家）                                │
│  ├─ 服务端等待足够人数加入（当前=2人，3v3=6人）            │
│  ├─ 客户端显示"等待对手加入... (1/2)"                      │
│  └─ 人数到齐 → 进入 HERO_SELECT                           │
├──────────────────────────────────────────────────────────┤
│  阶段2: HERO_SELECT（英雄选择）                            │
│  ├─ 服务端分配阵营（红队/蓝队，沿用现有 Team 系统）        │
│  ├─ 客户端弹出英雄选择界面（复用 HeroSelectUI）            │
│  ├─ 每个玩家选择英雄 → 通知服务端锁定                      │
│  ├─ 倒计时 30 秒，超时自动随机分配                          │
│  └─ 全部锁定 → 进入 LOADING                                │
├──────────────────────────────────────────────────────────┤
│  阶段3: LOADING（加载准备）                                │
│  ├─ 服务端: 为每个玩家生成角色 + 装备技能                   │
│  ├─ 客户端: 显示双方英雄展示（VS 画面）                     │
│  ├─ 等待所有玩家角色就绪                                    │
│  └─ 全部就绪 → 进入 BATTLE                                 │
├──────────────────────────────────────────────────────────┤
│  阶段4: BATTLE（对战进行中）                               │
│  ├─ 3秒倒计时后正式开战                                    │
│  ├─ 沿用现有战斗系统（技能/普攻/移动）                      │
│  ├─ 击杀计数、死亡重生沿用现有逻辑                          │
│  └─ 达到胜利条件 → 进入 RESULT                              │
├──────────────────────────────────────────────────────────┤
│  阶段5: RESULT（结算）                                     │
│  ├─ 服务端: 计算战绩数据（击杀/死亡/伤害/助攻）             │
│  ├─ 客户端: 显示结算界面（胜负 + MVP + 数据统计）           │
│  ├─ 10秒后显示按钮: "再来一局" / "离开"                     │
│  └─ 选择再来一局 → 回到 WAITING                             │
│  └─ 选择离开 → 断开连接                                     │
└──────────────────────────────────────────────────────────┘
```

### 2.2 M1 — 匹配与就绪系统

**2.2.1 核心机制**

服务端维护一个**游戏状态机**（GameState），通过 RemoteEvent 广播状态变化给所有客户端。

| 状态 | 进入条件 | 退出条件 | 持续时长 |
|------|---------|---------|---------|
| `WAITING` | 服务器启动 / 上一局结束 | 在线玩家数 ≥ 所需人数 | 无限等待 |
| `HERO_SELECT` | 人数到齐 | 所有玩家锁定英雄 或 倒计时结束 | 最长 30 秒 |
| `LOADING` | 英雄选择完成 | 所有玩家角色已生成 | 最长 10 秒 |
| `BATTLE` | 加载完成 | 达到胜利条件 | 无上限 |
| `RESULT` | 对战结束 | 倒计时结束（15秒） | 15 秒 |

**2.2.2 玩家加入/离开处理**

| 事件 | 处理方式 |
|------|---------|
| 玩家在 WAITING 加入 | 加入等待队列，更新人数广播 |
| 玩家在 HERO_SELECT 离开 | 取消当局，所有人回到 WAITING |
| 玩家在 BATTLE 离开 | 对手自动获胜，进入 RESULT |
| 玩家在 RESULT 离开 | 正常离开，不影响他人 |
| 玩家在 BATTLE 断线重连 | Roblox 自动重连机制，角色重新生成，保留队伍和击杀数 |

**2.2.3 阵营分配规则**

沿用现有 `RedTeam`/`BlueTeam` 机制，但改为**在 HERO_SELECT 阶段开始时一次性分配**（而非现在的玩家加入时立即分配）。

分配算法：
1. 所有等待中的玩家按加入顺序排列
2. 交替分配到红队/蓝队（1号→红，2号→蓝，3号→红...）
3. 3v3时：1/3/5→红队，2/4/6→蓝队

### 2.3 M2 — 英雄选择

**2.3.1 英雄选择流程**

```
HERO_SELECT 开始（服务端广播）
    ↓
客户端收到 → 显示英雄选择UI
    ├─ 展示所有可选英雄（当前4个）
    ├─ 实时显示对手是否已锁定（不显示选了谁）
    ├─ 右下角 30 秒倒计时
    ↓
玩家点击英雄 → 选中状态（高亮）
    ↓
玩家点击"锁定" → FireServer 通知服务端
    ├─ 服务端记录选择
    ├─ 广播"玩家X已锁定"
    ↓
全部锁定（或倒计时结束）
    ├─ 超时未选择的玩家 → 服务端随机分配
    └─ 广播进入 LOADING
```

**2.3.2 与现有 HeroSelectUI 的关系**

现有 `HeroSelectUI` 是客户端独立的选人流程，改造点：
- **新增**: 服务端控制选人开始/结束时机（而非客户端自主弹出）
- **新增**: 锁定机制（选择后确认，而非选了就确定）
- **新增**: 倒计时
- **新增**: 显示对方锁定状态
- **保留**: 英雄列表展示、英雄预览

### 2.4 M3 — 对战流程

**2.4.1 战斗开始**

LOADING → BATTLE 切换时：
1. 服务端为每个玩家在各自出生点生成角色
2. 红队出生点A，蓝队出生点B（现有地图两端）
3. 广播"对战即将开始"
4. 3秒倒计时（3...2...1...GO!）
5. 倒计时期间禁止移动和技能释放
6. 倒计时结束 → 解锁控制

**2.4.2 战斗进行中**

完全沿用现有系统，无需改动：
- 移动系统（MovementManager）
- 输入系统（InputManager）
- 技能系统（PlayerSkillManager + 15个Skill_XXXX）
- 普攻系统（AutoAttackManager）
- 伤害计算（CombatUtils）
- 死亡/重生（MatchSystem 现有逻辑）
- 头顶UI（OverheadUI 敌我区分）
- HUD击杀显示（UI_HUD）

**2.4.3 胜利条件**

沿用现有: 任一玩家达到 `KILLS_TO_WIN = 3` 杀 → 该玩家所在队伍获胜。

> 💡[假设] 3v3阶段可能改为"摧毁基地"模式，但当前2人阶段保持3杀胜利

**2.4.4 对战中断处理**

| 场景 | 处理 |
|------|------|
| 一方离开 | 剩余玩家自动获胜，直接进入 RESULT |
| 服务器无人 | 正常清理，重置回 WAITING |

### 2.5 M4 — 结算与重开

**2.5.1 结算数据**

| 数据项 | 来源 | 说明 |
|--------|------|------|
| 击杀数 | MatchSystem.killCount | 已有 |
| 死亡数 | 新增：服务端追踪 | 每次 Humanoid.Died += 1 |
| 造成伤害 | 新增：累计伤害统计 | 每次伤害 += damage |
| 承受伤害 | 新增：累计承受统计 | 每次被击 += damage |

**2.5.2 结算界面**

```
┌───────────────────────────────────────────────┐
│                  🏆 胜利！/ 💀 失败！           │
│                                                │
│   红队                     蓝队                │
│   [英雄头像]  VS  [英雄头像]                    │
│   玩家A                   玩家B                 │
│                                                │
│   ┌─────────┬──────┬──────┐                    │
│   │  数据    │ 玩家A │ 玩家B │                  │
│   ├─────────┼──────┼──────┤                    │
│   │ 击杀     │  3   │  1   │                    │
│   │ 死亡     │  1   │  3   │                    │
│   │ 造成伤害 │ 1250 │  820 │                    │
│   ├─────────┴──────┴──────┤                    │
│   │      MVP: 玩家A       │                    │
│   └───────────────────────┘                    │
│                                                │
│   [ 🔄 再来一局 ]    [ 🚪 离开 ]               │
└───────────────────────────────────────────────┘
```

**2.5.3 再来一局流程**

1. 玩家点击"再来一局" → 通知服务端
2. 服务端清理当局数据（击杀数/伤害统计/角色）
3. 回到 WAITING 状态
4. 等待所有玩家就绪（两人都点了"再来一局"）
5. 进入新一局的 HERO_SELECT

### 2.6 3v3 扩展性设计

**本次不实现但架构必须支持**:

| 扩展点 | 当前设计 | 3v3 需要的变化 |
|--------|---------|---------------|
| 玩家人数 | `REQUIRED_PLAYERS = 2` | 改为 `6` |
| 阵营分配 | 交替分配 2 人 | 交替分配 6 人（每队3人） |
| 英雄选择 | 同时选，允许重复 | 改为 Ban/Pick 顺序选 |
| 胜利条件 | 3杀胜利 | 改为摧毁基地 / 击杀数 |
| 结算界面 | 2人数据表 | 6人数据表 + 团队统计 |
| 出生点 | 2个（红/蓝各1） | 6个（红/蓝各3） |
| 匹配 | 同服自动 | TeleportService 跨服匹配 |

**架构要求**（给主程的指导方向，不做技术选型）:
1. 游戏状态机应为可配置的，状态和转换规则不硬编码
2. 所有"人数"相关逻辑使用配置变量，不用魔法数字
3. 阵营分配算法支持 N 队 × M 人
4. 结算数据结构是数组，不是固定2人的结构
5. RemoteEvent 的数据格式支持任意数量的玩家数据

## 第三章 验收标准

### 3.1 游戏状态机（M1）

- [ ] **AC-001**: 服务器启动后，游戏状态初始为 `WAITING`，客户端显示等待界面并显示当前在线人数（如"1/2"）
- [ ] **AC-002**: 当第2个玩家加入服务器时（在线人数 ≥ `REQUIRED_PLAYERS`），游戏状态自动从 `WAITING` 切换到 `HERO_SELECT`，所有客户端收到状态变更通知
- [ ] **AC-003**: 游戏状态按 `WAITING → HERO_SELECT → LOADING → BATTLE → RESULT` 顺序流转，不可跳过任何阶段
- [ ] **AC-004**: 在 `HERO_SELECT` 阶段，如果一个玩家断开连接（在线人数 < `REQUIRED_PLAYERS`），游戏状态回退到 `WAITING`，剩余玩家收到"对手已离开"提示
- [ ] **AC-005**: 在 `BATTLE` 阶段，如果一个玩家断开连接，剩余玩家自动判定获胜，游戏状态切换到 `RESULT`

### 3.2 英雄选择（M2）

- [ ] **AC-006**: 进入 `HERO_SELECT` 后，两个玩家同时看到英雄选择界面，显示所有可用英雄和 30 秒倒计时
- [ ] **AC-007**: 玩家选择并锁定英雄后，服务端记录该玩家的选择，并广播"该玩家已锁定"状态给对方客户端（不透露具体英雄）
- [ ] **AC-008**: 当所有玩家都锁定英雄后，游戏状态立即从 `HERO_SELECT` 切换到 `LOADING`，无需等待倒计时结束
- [ ] **AC-009**: 30 秒倒计时结束时，未锁定英雄的玩家由服务端随机分配一个英雄，然后正常进入 `LOADING`
- [ ] **AC-010**: 两个玩家可以选择相同的英雄，不做限制

### 3.3 加载与对战开始（M3）

- [ ] **AC-011**: 在 `LOADING` 阶段，服务端在红队出生点和蓝队出生点分别生成两个玩家的角色，并自动装备对应英雄的技能
- [ ] **AC-012**: `LOADING` 切换到 `BATTLE` 时，客户端显示 3 秒倒计时（3...2...1...GO!），倒计时期间玩家不能移动、释放技能或普攻
- [ ] **AC-013**: 倒计时结束后，两个玩家的移动和技能操作同时解锁

### 3.4 对战进行中

- [ ] **AC-014**: 在两台不同电脑上，玩家A 的移动操作在玩家B 的屏幕上实时可见（延迟 < Roblox 原生网络延迟 + 100ms）
- [ ] **AC-015**: 在两台不同电脑上，玩家A 对玩家B 释放技能时，伤害由服务端计算并同步到双方客户端，双方看到的血量一致
- [ ] **AC-016**: 击杀发生时，双方客户端都显示击杀播报，击杀计数正确更新
- [ ] **AC-017**: 玩家死亡后，5 秒重生倒计时在死亡玩家的客户端显示，重生后角色出现在己方出生点
- [ ] **AC-018**: 任一玩家达到 `KILLS_TO_WIN`（默认3）杀时，游戏状态从 `BATTLE` 切换到 `RESULT`

### 3.5 结算（M4）

- [ ] **AC-019**: 进入 `RESULT` 后，双方客户端都显示结算界面，包含：胜负结果、双方英雄、击杀数、死亡数
- [ ] **AC-020**: 结算界面上显示"再来一局"和"离开"按钮
- [ ] **AC-021**: 两个玩家都点击"再来一局"后，游戏状态回到 `WAITING`（如果人数仍然 ≥ `REQUIRED_PLAYERS`则直接进入 `HERO_SELECT`）
- [ ] **AC-022**: 玩家点击"离开"后，该玩家断开连接/离开游戏

### 3.6 边界与异常

- [ ] **AC-023**: `REQUIRED_PLAYERS` 为可配置变量（当前=2），修改为 6 后系统应等待 6 人才开始
- [ ] **AC-024**: 在 `WAITING` 阶段，只有1个玩家时，该玩家可以正常在大厅中看到等待UI，不会因为缺人导致脚本报错
- [ ] **AC-025**: 在 `RESULT` 阶段，如果一个玩家已离开，另一个玩家点击"再来一局"后应回到 `WAITING` 等待新对手
- [ ] **AC-026**: 快速连续点击"锁定英雄"按钮不会导致重复发送锁定请求或报错
- [ ] **AC-027**: 在 `LOADING` 阶段角色生成失败（如英雄配置缺失），服务端应能处理错误而不崩溃，并广播错误状态

## 第四章 风险与依赖

### 4.1 技术风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| Roblox 网络延迟导致技能不同步 | 中 | 高 | 当前技能系统已是"客户端发请求→服务端执行→广播结果"模式，天然适合联网。关注延迟感知 |
| 英雄选择状态不同步 | 低 | 中 | 服务端权威，客户端只展示服务端广播的状态 |
| 断线重连后状态丢失 | 中 | 中 | Roblox 自动重连会重新执行 Client.client.lua，需确保重连后能恢复到正确的游戏状态 |
| CharacterAutoLoads 与新流程冲突 | 低 | 高 | 当前已禁用 CharacterAutoLoads（MatchSystem），需确保新流程正确控制角色生成时机 |

### 4.2 设计风险

| 风险 | 缓解措施 |
|------|---------|
| 等待时间过长（只有2人但迟迟不来第2个） | 显示等待状态 + 允许单人练习模式（未来） |
| 英雄选择30秒太短/太长 | 可配置参数，后续根据测试调整 |
| 结算界面数据不准确 | 服务端权威统计，客户端只展示 |
