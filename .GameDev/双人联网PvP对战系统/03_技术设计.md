# REQ-002: 双人联网PvP对战系统 — 技术设计

## 1. 策划案评审

### 1.1 技术可行性评估

| 策划功能 | 可行性 | 说明 |
|---------|--------|------|
| 游戏状态机 (5状态) | ✅ 完全可行 | 服务端 Luau 状态机 + RemoteEvent 广播 |
| 等待对手加入 | ✅ 完全可行 | PlayerAdded 事件 + 人数检查 |
| 同步英雄选择 | ✅ 可行，需改造 | 现有 HeroSelectUI 是纯客户端，需增加服务端协调 |
| 对战倒计时 | ✅ 完全可行 | 服务端 task.delay + RemoteEvent 广播 |
| 结算界面 | ✅ 完全可行 | 服务端收集数据 → RemoteEvent → 客户端渲染 |
| 断线处理 | ✅ 可行 | PlayerRemoving 事件 + 状态回退 |
| 3v3扩展性 | ✅ 架构预留 | 使用配置变量 + 数组结构 |

### 1.2 关键发现（代码审查）

| 发现 | 影响 | 处理方式 |
|------|------|---------|
| `HeroConfig.lua` 内容异常 | ⚠️ 可能影响英雄选择逻辑 | 在 Roblox Studio 中确认实际数据。代码中 `HeroConfig[selectedHeroID]` 需要字符串键格式。**如果 Studio 中有正确数据则不改，否则需修复** |

| 英雄选择纯客户端 | ⚠️ 核心问题 | 增加 RemoteEvent 协调选人流程 |

### 2.2 设计决策：新建 GameManager vs 改造 MatchSystem

**方案A**: 直接改造 MatchSystem（在现有183行基础上扩展）
- ✅ 改动文件少
- ❌ MatchSystem 职责膨胀（匹配+选人+战斗+结算），单一职责违反
- ❌ 混合"流程管理"和"战斗逻辑"，难以维护

**方案B ★ 采用**: 新建 GameManager + 精简 MatchSystem
- ✅ 职责分离：GameManager管流程，MatchSystem管战斗
- ✅ 3v3扩展时只改GameManager的配置和选人逻辑
- ✅ MatchSystem的击杀/重生逻辑已经稳定，不需要动
- ❌ 多一个文件

**结论**: 采用方案B。GameManager 是新的"流程总控"，MatchSystem 被精简为"战斗辅助"。

### 2.3 GameState 状态机设计

```lua
-- 状态定义
local GameState = {
    WAITING     = "WAITING",      -- 等待玩家
    HERO_SELECT = "HERO_SELECT",  -- 英雄选择
    LOADING     = "LOADING",      -- 加载准备
    BATTLE      = "BATTLE",       -- 对战中
    RESULT      = "RESULT",       -- 结算
}

-- 合法转换
local TRANSITIONS = {
    [GameState.WAITING]     = { GameState.HERO_SELECT },
    [GameState.HERO_SELECT] = { GameState.LOADING, GameState.WAITING },  -- WAITING = 有人离开
    [GameState.LOADING]     = { GameState.BATTLE },
    [GameState.BATTLE]      = { GameState.RESULT },
    [GameState.RESULT]      = { GameState.WAITING },
}
```

### 2.4 服务端 ↔ 客户端通信协议

#### 新增 RemoteEvent

| 事件名 | 方向 | 数据格式 | 触发时机 |
|--------|------|---------|---------|
| `GameStateEvent` | Server→All | `{ state: string, data: table }` | 状态切换时 |
| `HeroSelectEvent` | 双向 | Client→Server: `{ heroId: string }` / Server→All: `{ playerId: number, locked: bool }` | 英雄选择/锁定 |
| `BattleStartEvent` | Server→All | `{ countdown: number }` | LOADING→BATTLE |
| `MatchResultEvent` | Server→All | `{ winner: string, players: [{name, hero, kills, deaths, damage}] }` | BATTLE→RESULT |
| `RematchEvent` | 双向 | Client→Server: `{ action: "rematch" \| "leave" }` / Server→All: `{ playerId, ready }` | RESULT 阶段 |

#### 保留 RemoteEvent (不改动)

| 事件名 | 说明 |
|--------|------|
| `MatchStateEvent` | 保留用于 BATTLE 阶段的击杀更新广播 |
| `DeathTimerEvent` | 保留用于死亡倒计时 |
| `CastSkillEvent` / `EquipSkillEvent` 等 | 完全不动 |

## 4. 任务清单（按执行顺序）

### Phase 1: 服务端核心（GameManager + MatchSystem改造）

**T1: GameManager.server.lua — 游戏状态机与流程控制**
- 新建文件: `src/ServerScriptService/GameManager.server.lua`
- 实现: GameState 状态机、状态转换验证、RemoteEvent 创建与广播
- 实现: WAITING 阶段逻辑（等待人数到齐）
- 实现: HERO_SELECT 阶段（阵营分配、倒计时、锁定协调、超时随机）
- 实现: LOADING 阶段（角色生成、技能装备）
- 实现: BATTLE→RESULT 转换（接收MatchSystem的胜负通知）
- 实现: RESULT 阶段（数据收集、重开/清理）
- 实现: 玩家加入/离开的各状态处理
- AC覆盖: AC-001~005, AC-008~011, AC-018, AC-021~025, AC-027~029
- 预估: ~350行

**T2: MatchSystem.server.lua — 精简改造**
- 移除: 阵营分配逻辑（`assignTeam`）、`onPlayerAdded` 中的 `LoadCharacter()` 调用
- 改造: 暴露 API 供 GameManager 调用（`StartBattle()`, `EndBattle()`, `ResetMatch()`）
- 保留: 击杀计数、死亡重生、`checkWinCondition`、`broadcastMatchState`
- 新增: 死亡数和伤害统计追踪
- 新增: 胜负判定后回调 GameManager（通过 BindableEvent 或直接 require）
- AC覆盖: AC-016~018
- 预估: ~80行改动

### Phase 2: 客户端改造

**T3: Client.client.lua — 状态机驱动改造**
- 改造: 从"阻塞式 HeroSelectUI.Show()"改为"监听 GameStateEvent 的状态驱动流程"

- 新增: 30秒倒计时显示
- 新增: 对方锁定状态显示（"对手已锁定"文字）
- AC覆盖: AC-006~010, AC-026
- 预估: ~80行改动

**T5: UI_HUD.lua — 等待/倒计时/结算UI**

## 6. 技术风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| MatchSystem 和 GameManager 初始化竞争 | 中 | 高 | GameManager 在初始化时等待 MatchSystem 就绪（通过 Attribute 或 BindableEvent） |
| HeroConfig.lua 内容异常 | 低 | 中 | 代码中兼容检查，如果 Config 缺失则给默认英雄 |
| 客户端 GameStateEvent 丢失 | 低 | 高 | 新玩家加入时，GameManager 补发当前状态 |
| RESULT→WAITING 清理不彻底 | 中 | 中 | 明确清理清单：销毁角色、重置 killCount、清空 heroId |
